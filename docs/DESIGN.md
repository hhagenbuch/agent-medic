# agent-medic — Design (RFC)

**Status:** Draft / pre-code (Phase 0)
**Author:** Heyward Hagenbuch

The immune system for agents: a closed loop from production failure →
regression case → proposed fix → gated canary → human-approved promotion.
Every step is an existing, tested primitive from this portfolio; medic is the
orchestration and — more importantly — the **authority boundaries** between
them. This document is the contract; code follows it in phases.

## 1. Problem

When an agent fails in production today, a human does everything: notices the
failure, reads the trace, writes a regression test (usually never), edits the
prompt, ships it, and hopes. Each step exists as a tool in this portfolio —
[agent-blackbox](https://github.com/hhagenbuch/agent-blackbox) records and
exports failures, [agent-evals](https://github.com/hhagenbuch/agent-evals)
gates quality, [agent-operator](https://github.com/hhagenbuch/agent-operator)
canaries and promotes prompt changes — but the seams between them are manual.

medic closes the loop, with one deliberate exception: **a human holds exactly
one button — approve.** The interesting design work is not the plumbing; it is
deciding which stages may use an LLM, what the repair agent is allowed to
touch, and when the machine must admit defeat.

The payoff line: **the system's response to failure is to grow an antibody.**
Every incident becomes a permanent eval case; the suite only ever gets
stronger.

## 2. The loop

```
   [Watcher]              [Diagnoser]                 [Surgeon]
blackbox traces  ──►  failing trajectory  ──►  repair agent (starter + MCP)
 (live tail)          export-eval case            proposes a prompt fix
                                                        │
   [Gate]                                               ▼
agent-evals suite  ◄──  agent-operator canary  ◄──  MedicProposal CR
(incident case + full suite must pass)                  │
        │ pass                                          │ fail
        ▼                                               ▼
  await HUMAN APPROVAL → Promoted            RolledBack + report attached
```

One Spring Boot app (`agent-medic`) + one CRD (`MedicProposal`). The app hosts
the Watcher, the Diagnoser, the medic MCP server, the embedded Surgeon, and
the JOSDK reconciler.

## 3. Components and their authority

### 3.1 Watcher — rules-as-data, no LLM

Tails the blackbox trace directory (a shared volume in-cluster; a plain
directory locally). Flags failing trajectories by **declarative rules**:

- `error` events in the trace
- honesty violations (e.g. claimed-sent-but-queued: an `assistant_message`
  asserting an action whose `tool_call` never appears or whose `tool_result`
  shows deferral)
- tool-discipline breaks (spurious or missing tool calls per rule)
- eval-sampled failures (a scheduled agent-evals run against prod fails a case)

Rules are data (YAML), unit-tested table-driven. A firing rule produces an
**incident**: `{traceRef, turn, rule, excerpt}`.

### 3.2 Diagnoser — deterministic, no LLM

Runs `blackbox export-eval` on the failing turn to produce an agent-evals case
(the `EvalExporter` from blackbox-cli, used as a library), then assembles the
**incident bundle**: the exported case, the full trace, the divergence report
if a replay was run, and the provoking input.

**This stage must never hallucinate evidence, so it contains no LLM.** The
bundle is what the Surgeon reads, what the gate replays, and what the human
reviews — if it were generated by a model, the whole chain of custody would be
poisoned. This is the first of the three authority decisions.

### 3.3 Surgeon — an agent with a one-line permission slip

The repair agent. It runs on
[spring-ai-agent-starter](https://github.com/hhagenbuch/spring-ai-agent-starter)'s
`AgentLoop` (embedded in the medic app as a library — the same consumption
pattern agent-meter's starter adapter proved), and its tools come from the
starter's own MCP client (`McpClient`/`McpToolAdapter`) connected to a
**read-only medic MCP server** exposing exactly three tools:

| tool | returns |
|---|---|
| `get_incident` | the incident bundle (case, trace excerpt, rule, divergence) |
| `get_current_prompt` | the agent's active system prompt (from the live `PromptVersion`) |
| `get_eval_history` | recent gate reports for this agent (pass rates, failing case ids) |

The Surgeon's output is a **proposal**: a full replacement system prompt plus
a written rationale, returned as a JSON document. Model output is untrusted
input: the proposal is schema-validated, size-capped, and diffed against the
current prompt before anything touches the cluster.

**Authority is structural, not conventional** (the second authority decision):
the proposal schema *has no field* for tools, model, code, or config — the
controller builds the `PromptVersion` itself and copies `model` and everything
else from the current `Agent` spec. A Surgeon that "wants" to change a tool
has no channel through which to say so. Prompts only, in the MVP and by
design.

The Surgeon never sees secrets — the MCP tools serve the incident bundle
(already redaction-filtered by blackbox-core at record time), the prompt, and
eval history; nothing else. And the medic is subject to the same observability
as its patients: the Surgeon's own runs are traced by blackbox and metered by
[agent-meter](https://github.com/hhagenbuch/agent-meter).

### 3.4 MedicProposal CRD + controller

`medic.hhagenbuch.io/v1alpha1`, sitting beside the operator's
`agents.hhagenbuch.io` group:

```yaml
apiVersion: medic.hhagenbuch.io/v1alpha1
kind: MedicProposal
metadata:
  name: support-agent-20260721-1432
spec:
  agentRef: support-agent
  incident:
    traceRef: traces/support-agent/s-8f3a.trace.jsonl
    turn: 3
    rule: honesty.claimed-sent-but-queued
    caseConfigMap: support-agent-20260721-1432-case
  maxAttempts: 2            # then NeedsHuman
status:
  phase: Proposing          # Proposing → Gating → AwaitingApproval → Promoted
                            #                    ↘ RolledBack(attempt++) ↘ NeedsHuman
  attempts:
    - promptVersionRef: support-agent-medic-1432-a1
      rationale: "The prompt instructs 'confirm the email was sent'..."
      evalPassRate: "0.82"
      incidentCasePassed: false
      outcome: RolledBack
```

Reconcile flow:

1. **Proposing** — run the Surgeon against the incident; validate the
   proposal; record diff + rationale in status.
2. **Gating** — build a **candidate dataset ConfigMap** = current suite +
   the incident case, and create a `PromptVersion` (strategy
   `EvalGatedCanary`) pointing its gate at the candidate dataset. The
   operator's canary machinery does the rest — medic does not reimplement it.
   Note: `evalGate.datasetConfigMap` currently lives on `Agent.spec`, so
   this requires a per-PromptVersion `evalGateOverride` — the second of the
   two operator extensions (§3.5). Medic patching `Agent.spec.evalGate`
   instead is rejected: it walks into the spec-ownership footgun the
   operator's own design calls out, and races with humans editing the Agent.
3. The gate must clear **two bars**: the operator's aggregate
   `minPassRate`, **and** the incident case itself must pass. An aggregate
   0.9 can be met while the one case that started all this still fails.
   Both bars are enforced natively by agent-evals (≥ 0.2.0): the merged
   incident case carries `required: true` (when stable — see the autoimmune
   guard, §5), so the eval Job itself exits 1 on it regardless of the
   aggregate. Medic reads the outcome from the Job log's final
   `VERDICT-JSON: {...}` line (schema `agent-evals/verdict/v1`) — the pod
   log is the one channel the operator already exposes, needing no shared
   volume, extra RBAC, or sidecar, and the tagged schema-versioned line
   survives human-format drift that scraping `[PASS]` lines would not.
   Fail-closed: a completed log with no verdict line, an unparseable
   verdict, or a verdict without the incident case is a broken gate, never
   a pass. Medic pins the gate's evals image via `evalGateOverride.image`
   so the verdict contract is guaranteed present.
4. **Pass → AwaitingApproval.** Promotion is held for a human. Approval is
   an annotation on the MedicProposal — `medic.hhagenbuch.io/approved: "true"`
   — applied by a person who has the trace, the diff, the rationale, and the
   eval report in front of them (all linked from status). Auditable,
   RBAC-controllable, `kubectl describe`-visible.
5. **Approved → Promoted** — the operator promotes; medic then merges the
   incident case into the **permanent** suite ConfigMap. This is the
   antibody rule: cases merge on promotion and are never deleted or
   weakened. A future fix that regresses this case is rejected by
   construction, because the gate always runs the whole suite.

   Ownership: the permanent suite ConfigMap is **medic-owned** (labeled
   `medic.hhagenbuch.io/managed: "true"`) so the antibody merge is not
   silent GitOps drift. Humans keep adding cases via git as usual; a merge
   step folds the git-sourced cases and the accumulated antibodies into the
   ConfigMap deterministically (antibodies win on id collision — never
   weakened). Write-back of antibodies to git as an automated PR is
   roadmap, not MVP.
6. **Fail → RolledBack** — the canary is torn down (operator behavior),
   the failing report lands in MedicProposal status, and the Surgeon gets
   one more attempt with the failure report added to the incident bundle.
   After `maxAttempts` (default 2): **NeedsHuman**, with the full evidence
   bundle. Two failed repairs means the machine is out of its depth — say so.

### 3.5 The approval hold — where it lives and why

The operator today auto-promotes on gate pass; medic must hold promotion for
approval. Two candidate designs:

- **(a) Operator-side hold** — extend `PromptVersion` with
  `spec.requireApproval` and an `AwaitingApproval` phase between gate-pass
  and `Promoted`. Enforcement lives in the thing that deploys; nothing can
  promote around it.
- **(b) Medic-side hold** — medic runs its own pre-approval gate and only
  creates the `PromptVersion` after a human approves; the operator then
  independently re-gates and auto-promotes.

**MVP choice: (a).** The approval is the product's safety story; a safety
control belongs at the enforcement point, not in the component that asks
politely. (b) leaves a window where anyone (or any bug) that can create a
`PromptVersion` ships an unapproved prompt — the hold would be a convention,
and this design rejects authority-by-convention on principle (§3.3). The cost
is a small, separately-reviewed feature PR to agent-operator carrying **two
generic, medic-agnostic extensions**: (1) `requireApproval` + an
`AwaitingApproval` phase + a release-annotation check, and (2) a
per-PromptVersion `evalGateOverride` (needed for the candidate dataset,
§3.4.2). Honest sequencing: the operator grows features any
human-in-the-loop workflow can use; medic stays a consumer.

### 3.6 No auto-approve mode. Ever. (the third authority decision)

There is no flag, env var, or annotation that skips the human. This is
documented as the safety story, not a missing feature: medic writes prompts
with an LLM, gates them with an LLM-judged suite, and deploys them to
production — a fully autonomous version of that loop is a prompt-injection
amplifier with a deploy badge. The human approval is where the loop's
authority terminates, and it is the reason every other stage is allowed to be
autonomous.

## 4. Design decisions (summary)

- **No LLM in the Diagnoser** — evidence must never be hallucinated; chain of
  custody from trace to gate is deterministic.
- **Prompts-only repair, enforced structurally** — the proposal schema cannot
  express any other change; the controller composes the PromptVersion.
- **No auto-approve, ever** — the single human button is the safety story.
- **Reuse the operator's canary machinery** — medic creates PromptVersions;
  it does not run deployments or eval Jobs itself.
- **Two-bar gate** — aggregate min-pass-rate AND the incident case by id.
- **Antibody rule** — incident cases merge into the permanent suite on
  promotion; medic never deletes or weakens a case.
- **Attempt cap → NeedsHuman** — bounded autonomy; failure is reported, not
  retried forever.
- **Surgeon embedded via the starter-as-library, tools via MCP** — dogfoods
  the starter's MCP client end-to-end; the medic MCP server is read-only.

## 5. Failure modes

- **Surgeon proposes nonsense** → schema validation rejects it (attempt
  consumed) or the gate fails it → attempt 2 → NeedsHuman. Nonsense never
  reaches production; worst case is wasted canary compute.
- **Fix passes the suite but not the incident case** → caught by the second
  bar (§3.4.3); the aggregate rate alone is explicitly not trusted.
- **Flaky incident case — the autoimmune guard.** A required antibody is a
  permanent veto over every future promotion; graft a flaky case in as
  `required` and the immune system starts attacking healthy deploys at
  random — an autoimmune disease. So the required flag must be EARNED at
  export time: the Diagnoser re-runs the fresh case 3× against the live
  agent (deterministic assertions only). Stable failure (0/3 passes) →
  `required` antibody. Anything flaky → the case is merged as ADVISORY
  (gate may pass without it), the probe record lands in the incident
  bundle, and an advisory failure at the gate is surfaced in the hold
  message for the human to weigh — never auto-vetoed, never suppressed.
  Probe unavailable (or file-only mode) → default `required`: the human
  hold still protects the suite, and an unverifiable case blocking a
  promotion is recoverable, while a flaky required antibody merged into
  the permanent suite is not.
- **Incident storm** (one bad deploy → hundreds of firing rules) → dedupe by
  `(agentRef, rule)` with an open-proposal cap of 1 per pair; subsequent
  incidents attach to the open MedicProposal as additional evidence instead
  of spawning proposals.
- **Watcher rule bug flags healthy behavior** → the gate + human approval
  bound the damage: a "fix" for a non-bug still has to beat the full suite
  and convince a person holding the evidence.
- **Wedged gate** → inherited from the operator (Job `activeDeadlineSeconds`
  + reconciler timeout → RolledBack), consumed by medic as a failed attempt.

## 6. Non-goals (MVP)

- **No tool, code, or model repair** — prompt text only.
- **No multi-incident batching** — one proposal per incident (with the storm
  dedupe above); no "fix these five things at once" prompts.
- **No auto-approve** — restated here so it appears in every list someone
  might read.
- No cross-agent learning (an antibody belongs to one agent's suite).
- No UI; `kubectl` + the status subresource are the interface.
- No multi-cluster; no webhook admission control (reconciler-side validation,
  same trade-off as the operator).

## 7. Tech + local target

- Java 25, Spring Boot 3.5.x, Maven — the current house stack (blackbox/meter
  level).
- JOSDK for the MedicProposal controller; Fabric8 `crd-generator-apt` for CRD
  YAML from Java sources (operator pattern).
- Dependencies consumed as libraries via JitPack: starter (AgentLoop, MCP
  client), blackbox-core/cli (trace reader, EvalExporter), evals report
  parsing.
- Local cluster: **kind**; everything runs on a laptop. `hack/` scripts per
  the operator's pattern.
- Tests: table-driven unit tests on Watcher rules; fake-LLM Surgeon tests
  (scripted proposal); kind e2e for the full loop; live CI smoke with the
  repo-secret key per the established live-smoke pattern.

## 8. Build phases

0. **Design** — this document, committed alone.
1. **Watcher + Diagnoser** — trace tailing, failure rules, export-eval
   integration, incident bundle format. e2e with a seeded bad trace.
2. **Surgeon** — medic MCP server (three read-only tools), repair prompt,
   proposal schema + validation. Fake-LLM tests; optional live test.
3. **MedicProposal controller** — CRD, reconciler reusing the operator's
   canary machinery, the operator's `requireApproval` feature PR, approval
   annotation flow, status wiring. kind e2e: seeded incident → proposal →
   gate fail (bad fix) → second proposal → gate pass → AwaitingApproval →
   annotate → Promoted → antibody present in next gate run.
4. **The demo** — scripted recording: sabotage a live agent's prompt →
   Watcher flags the honesty failure → case exported → Surgeon proposes →
   canary gate → approve → healed, suite one case stronger.
5. **`docs/STANDARDS.md`** — the public agent-engineering standards
   (orchestration/prompt separation, prompts-as-code,
   model-output-as-untrusted-input, two-reviewer rule for prompt/tool
   changes, eval gate as a pipeline stage). Medic hosts the canonical copy
   because it is the repo that changes prompts programmatically — the repo
   where the standards are load-bearing.

## 9. Guardrails

Clean room. No employer names, domains, code, or internal document text
anywhere — the standards doc in phase 5 is a from-scratch public writing of
transplantable doctrine, GitHub-Actions-flavored. CI greps for employer
strings (established pattern).
